## 렉시컬 스코프

- 렉시컬 스코프는 함수가 **어디서 호출됐는지**가 아니라 **어디서 선언됐는지**를 기준으로 상위 스코프가 결정되는 규칙이다.
- 즉, 함수가 만들어질 때 어떤 스코프를 바라볼지가 정해진다.

## 렉시컬 환경

- 렉시컬 환경은 실행 중인 코드에서 **변수와 함수 이름을 저장하고 관리하는 구조**다.
- 현재 스코프에 어떤 식별자가 있는지, 그리고 바깥 스코프와 어떻게 연결되는지를 함께 관리한다.
- 쉽게 말해, JavaScript가 변수를 찾을 때 참고하는 내부 정보 묶음이라고 보면 된다.

## 블록 스코프와 함수 스코프

- 스코프는 **변수에 접근할 수 있는 범위**를 뜻한다.
- 함수 스코프는 함수 단위로 범위가 정해진다.
- 블록 스코프는 `{}` 블록 단위로 범위가 정해진다.

| 선언 키워드 | 스코프 | 재할당 | 실무 사용 기준 |
| --- | --- | ---: | --- |
| `var` | 함수 스코프 | 가능 | 레거시 코드가 아니면 거의 사용하지 않음 |
| `let` | 블록 스코프 | 가능 | 값이 바뀌어야 할 때 사용 |
| `const` | 블록 스코프 | 불가능 | 기본 선택. 재할당하지 않는 값에 사용 |

## 스코프 체인

- 스코프는 함수의 중첩으로 계층 구조를 만든다.
- 변수를 참조할 때 현재 스코프에 없으면, 바깥 스코프로 한 단계씩 올라가며 찾는다.
- 이런 탐색 구조를 스코프 체인이라고 한다.

## 클로저

- 클로저는 함수가 자신이 선언될 때의 바깥 스코프를 기억하고, 나중에도 그 스코프의 값을 참조할 수 있는 현상이다.
- 함수가 바깥 환경을 복사해 들고 있는 것이 아니라, 바깥 렉시컬 환경을 참조하고 있는 것이다.
- 클로저는 상태 유지, 콜백 처리, 함수 팩토리, 정보 은닉 같은 패턴에서 자주 쓰인다.

## 호이스팅

- 호이스팅은 JavaScript가 코드를 실행하기 전에 선언을 먼저 처리하기 때문에, 선언문이 위로 끌어올려진 것처럼 보이는 현상이다.
- 실제로 코드가 이동하는 것은 아니고, 실행 전에 필요한 선언 정보가 먼저 준비되는 것이다.

### 변수 호이스팅

- `var`, `let`, `const` 모두 호이스팅된다.
- 차이는 초기화 시점이다.
- `var`는 선언과 함께 `undefined`로 초기화된다.
- `let`, `const`는 선언은 먼저 되지만, 실제 선언문을 만나기 전까지 초기화되지 않는다.
- 그래서 초기화 전 구간에서는 접근할 수 없고, 이 구간을 TDZ라고 한다.

### 함수 선언문 호이스팅

- 함수 선언문은 실행 전에 함수 이름과 함수 객체가 함께 준비된다.
- 그래서 선언 전에 호출해도 실행할 수 있다.

### Temporal Dead Zone

- TDZ는 `let`, `const`로 선언한 변수를 **초기화되기 전까지 접근할 수 없는 구간**이다.
- 이 구간에서 변수를 읽으면 에러가 난다.

## 함수 선언문과 함수 표현식

### 함수 선언문

- `function foo() {}`처럼 이름을 직접 선언하는 방식이다.
- 실행 전에 함수 객체까지 준비되므로 선언 전에 호출할 수 있다.

### 함수 표현식

- 함수를 변수에 할당하는 방식이다.
- 예: `const foo = function () {}`
- 함수 자체보다 변수가 먼저 만들어지고, 실제 함수는 할당 시점에 들어간다.
- 그래서 보통 할당이 끝난 뒤에만 호출할 수 있다.

## 값 캡처와 Stale Closure

- React 함수 컴포넌트는 렌더링될 때마다 다시 실행된다.
- 각 렌더링은 그 시점의 상태값을 기준으로 동작한다.
- 렌더링 중 만들어진 이벤트 핸들러나 콜백은 **그때의 값**을 참조하는데, 이를 값 캡처라고 볼 수 있다.
- 예전 렌더에서 만들어진 함수가 계속 남아서 오래된 상태값을 참조하면, 이를 Stale Closure라고 한다.
- 이런 문제는 `setInterval`, `setTimeout`, 이벤트 리스너, 비동기 콜백에서 자주 나타난다.

### 함수형 업데이트

- 이전 상태를 기준으로 다음 상태를 계산할 때 사용한다.
- 오래된 클로저의 값이 아니라, React가 넘겨주는 최신 상태를 기준으로 계산할 수 있다.

```tsx
setCount((previousCount) => previousCount + 1);
```

### Effect 의존성 관리

- Effect 안에서 최신 상태값을 직접 사용해야 한다면, 그 상태를 의존성 배열에 넣어야 한다.
- 상태가 바뀌면 기존 Effect를 정리하고, 최신 값을 기준으로 Effect가 다시 실행된다.

```tsx
useEffect(() => {
  console.log(count);
}, [count]);
```

### 선택 기준

- 이전 상태를 바탕으로 다음 상태만 계산하면 된다면 **함수형 업데이트**를 사용한다.
- Effect의 동작 자체가 특정 상태값에 따라 달라진다면 **의존성 배열**로 관리한다.
- 다만 의존성으로 상태를 맞추는 방식은 Effect가 자주 다시 실행될 수 있으므로, 단순 누적 업데이트라면 함수형 업데이트가 더 적절한 경우가 많다.
